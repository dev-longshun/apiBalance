# 快速验证指南：Check Balance

**用途**: 验证功能端到端可用性的操作步骤

## 前置条件

- Go 1.22+ 已安装
- 至少一个 AI API 中转站的 base_url 和 api_key
- （可选）Telegram Bot Token 和 Chat ID（用于验证告警功能）

## 启动

```bash
# 克隆并构建
cd quota-sentinel
cp config.example.yaml config.yaml
# 编辑 config.yaml 填入 Telegram 凭证（可选）
go build -o quota-sentinel .
./quota-sentinel
```

系统启动后访问 `http://localhost:8080`。

## 验证场景

### 场景 1: 添加站点并查看余额

1. 打开 Web 面板，点击"添加站点"
2. 填入站点名称、URL、API Key
3. 提交后观察仪表盘
4. **预期**: 15 秒内显示站点卡片，余额和状态已填充

对应验收: [spec.md - 用户故事 1 - 场景 1](spec.md)

### 场景 2: 多格式自动探测

1. 添加一个 New API 站点（OpenAI 兼容格式）
2. 添加一个 sub2api 站点
3. **预期**: 两个站点均自动识别格式，返回正确余额，无需手动配置

对应验收: [spec.md - 用户故事 4](spec.md)

### 场景 3: 多级阈值告警

前提: Telegram 凭证已配置

1. 添加一个余额为 $45 的站点，设置阈值 [50, 10]
2. 等待第一次轮询（或手动点击"查询"）
3. **预期**: 收到 Telegram 消息 "余额不足提醒...触发阈值: $50.00"
4. 再次触发查询
5. **预期**: 不再收到 $50 的重复告警

对应验收: [spec.md - 用户故事 3 - 场景 1, 2](spec.md)

### 场景 4: 阈值回升重置

1. 沿用场景 3 的站点，假设用户已充值，余额回升至 $60
2. 等待一次轮询
3. 此后余额再次降至 $48
4. **预期**: 再次收到 $50 阈值的告警（阈值已因回升而重置）

对应验收: [spec.md - 用户故事 3 - 场景 4](spec.md)

### 场景 5: Telegram Bot 实时查询

前提: Telegram Bot 已启动

1. 在 Telegram 中发送 `/balance`
2. **预期**: Bot 回复"正在查询..."，随后返回所有站点的最新余额摘要
3. 发送 `/balance <某站点名称>`
4. **预期**: 返回该站点的详细余额信息，含阈值触发状态

对应验收: [spec.md - 用户故事 5 - 场景 1, 2](spec.md)

### 场景 6: 定时轮询

1. 在设置中将轮询间隔改为 5 分钟
2. 等待 5 分钟
3. **预期**: 所有站点的 "最后查询时间" 自动更新

对应验收: [spec.md - 用户故事 2 - 场景 1](spec.md)

### 场景 7: 错误处理

1. 添加一个 base_url 为不存在地址的站点
2. **预期**: 站点状态显示为红色"错误"，附带"连接超时"提示
3. 其他正常站点不受影响

对应验收: [spec.md - 用户故事 1 - 场景 2](spec.md)
