# 功能规格说明：Check Balance — AI 中转站额度监控与告警

**功能分支**: `001-check-balance`

**创建日期**: 2026-06-08

**状态**: 草稿

**输入**: 用户描述："一个轻量级 Web 应用，集中监控多家 AI API 中转站的剩余额度，自动探测不同的 API 格式，并在余额低于用户设定阈值时通过 Telegram 推送告警通知。"

## 澄清记录

### Session 2026-06-08

- Q: 告警防重复机制应该如何工作？ → A: 移除 6 小时冷却机制，改为多级阈值。用户可为每个站点设置多个阈值（如 $50、$10），每个阈值在余额首次低于时触发一次告警，余额回升到阈值之上后重新生效。
- Q: Telegram Bot 的 `/balance` 命令应返回缓存数据还是实时查询？ → A: 实时查询。用户发送 `/balance` 时，系统应立即查询所有渠道的最新余额并返回结果，而非仅显示缓存数据。

## 用户场景与测试 *(必填)*

### 用户故事 1 - 注册站点并查询余额 (优先级: P1)

作为 API 中转站用户，我希望添加一个上游站点（名称、URL、API Key），并立即看到该站点的当前余额，以便判断是否需要充值。

**优先级原因**: 这是最基础的交互——如果无法注册站点并查看余额，其他功能都没有意义。

**独立测试**: 通过 Web 面板添加一个站点，触发余额查询，验证余额和状态正确显示在仪表盘上。

**验收场景**:

1. **假设** 用户在空白仪表盘上，**当** 添加一个有效的站点（名称、URL、API Key），**则** 系统创建站点、立即查询余额，并在 15 秒内将结果显示在仪表盘上。
2. **假设** 用户添加了一个无法连接的 URL，**当** 余额查询执行时，**则** 该站点显示"错误"状态，并附带可读的错误信息（如"连接超时"）。
3. **假设** 用户添加了一个 API 格式未知的站点，**当** 系统尝试所有已知的探测方式均失败后，**则** 站点显示"错误"状态，提示"无法识别 API 格式"，而非崩溃或挂起。

---

### 用户故事 2 - 自动轮询余额 (优先级: P1)

作为 API 中转站用户，我希望系统在后台定时检查所有站点的余额，这样我就不用手动刷新。

**优先级原因**: 核心价值所在——无需人工干预的持续监控。

**独立测试**: 设置轮询间隔，等待一个周期，验证所有站点余额已刷新。

**验收场景**:

1. **假设** 已注册 5 个站点且轮询间隔设为 30 分钟，**当** 30 分钟过后，**则** 所有 5 个站点的余额被重新查询，仪表盘数据更新。
2. **假设** 系统刚启动完成，**当** 初始化结束后，**则** 在首次定时轮询之前，立即执行一轮全量余额查询。
3. **假设** 某个站点的响应时间超过 10 秒，**当** 轮询周期运行时，**则** 其他站点不受阻塞，结果照常显示。

---

### 用户故事 3 - 多级阈值 Telegram 告警 (优先级: P1)

作为 API 中转站用户，我希望为每个站点设置多个告警阈值（如 $50、$10），当余额首次低于某个阈值时通过 Telegram 通知我，这样我可以根据紧急程度分级响应。

**优先级原因**: 这是用户部署本系统的首要原因——多级告警比单一阈值更灵活，让用户在余额逐步下降的过程中获得分阶段提醒，避免错过充值时机。

**独立测试**: 为某站点设置两个阈值（$50 和 $10），让余额从 $80 逐步下降，验证在穿越 $50 和 $10 时各收到一次告警。

**验收场景**:

1. **假设** 某站点设置了阈值 [$50, $10]，当前余额为 $80，**当** 余额查询返回 $45，**则** 系统发送一条 Telegram 告警，提示余额已低于 $50 阈值。
2. **假设** 某站点已触发过 $50 阈值告警，当前余额为 $45，**当** 下一次查询仍返回 $45，**则** 不发送重复告警（该级别已触发）。
3. **假设** 某站点已触发过 $50 阈值告警，当前余额为 $45，**当** 余额查询返回 $8，**则** 系统发送一条新的 Telegram 告警，提示余额已低于 $10 阈值。
4. **假设** 某站点的 $50 阈值已触发，用户充值后余额回升至 $60，**当** 余额再次降至 $48，**则** $50 阈值重新生效，再次发送告警。
5. **假设** 未配置 Telegram Bot Token 或 Chat ID，**当** 检测到余额低于阈值，**则** 告警记录在本地日志中，不尝试发送 Telegram 消息，也不抛出错误。

### 用户故事 4 - 多格式自动探测 (优先级: P1)

作为 API 中转站用户，我希望系统自动识别上游站点使用的 API 格式，而不需要我手动配置技术细节。

**优先级原因**: API 中转站框架的多样性是核心技术挑战。没有自动探测，用户就需要手动研究每个站点的 API 格式，这违背了产品的初衷。

**独立测试**: 注册运行不同中转框架的站点（New API、sub2api 等），验证所有站点均能返回有效余额数据，无需任何逐站格式配置。

**验收场景**:

1. **假设** 站点运行 New API / One API（OpenAI 兼容格式），**当** 查询余额时，**则** 系统通过 `/v1/dashboard/billing/subscription` 探测到格式，返回正确的 USD 余额。
2. **假设** 站点运行 sub2api，**当** 查询余额时，**则** 系统通过 `/v1/usage` 探测到格式，返回正确单位的余额。
3. **假设** 站点使用 JWT 认证的 `/api/v1/auth/me`，**当** 查询余额时，**则** 系统探测到此格式并从响应中提取余额。
4. **假设** 站点使用 NewAPI Token 格式，**当** 查询余额时，**则** 系统通过 `/api/usage/token/` 探测到此格式，返回可用 Token 数量。

---

### 用户故事 5 - Telegram Bot 实时交互查询 (优先级: P1)

作为移动端用户，我希望通过 Telegram Bot 发送 `/balance` 命令时，系统能实时查询所有渠道的最新余额并返回结果，这样我无需打开 Web 面板就能随时获取最新数据。

**优先级原因**: 用户在移动端的核心交互方式。实时查询确保返回的是最新数据而非可能过期的缓存，对需要即时决策的场景（如判断是否需要立即充值）至关重要。

**独立测试**: 向 Bot 发送 `/balance`，验证系统触发实时查询并返回所有站点的最新余额摘要。

**验收场景**:

1. **假设** Bot 已配置且注册了 3 个站点，**当** 用户发送 `/balance`，**则** 系统立即对所有站点发起实时余额查询，查询完成后回复一份包含每个站点名称、最新余额和状态指示符的摘要。
2. **假设** 用户发送 `/balance Example API`，**当** 存在该名称的站点时，**则** 系统对该站点发起实时查询，回复该站点的详细最新信息。
3. **假设** 用户发送 `/balance`，**当** 某站点实时查询超时或失败，**则** 该站点在结果中标记为"查询失败"，其余站点的结果正常返回。
4. **假设** 用户发送 `/refresh`，**当** 收到时，**则** 系统立即触发一轮全量查询，更新仪表盘数据并回复确认消息。
5. **假设** 用户发送 `/help`，**当** 收到时，**则** Bot 回复可用命令列表及简短说明。

---

### 用户故事 6 - Web 面板站点管理 (优先级: P2)

作为 API 中转站用户，我希望通过 Web 面板编辑和删除现有站点、调整告警阈值、配置系统设置。

**优先级原因**: 日常管理的必要功能，但优先级低于最初的添加-查询流程。

**独立测试**: 编辑站点阈值、删除站点、调整轮询间隔——验证每项变更都能持久化并生效。

**验收场景**:

1. **假设** 站点已存在，**当** 用户将其告警阈值从 [$50, $10] 修改为 [$30, $5]，**则** 后续余额检查使用新阈值列表。
2. **假设** 站点已存在，**当** 用户删除该站点，**则** 它从仪表盘上移除，不再参与轮询。
3. **假设** 用户将轮询间隔从 30 分钟改为 15 分钟，**当** 保存后，**则** 调度器立即采用新间隔。
4. **假设** 用户输入 Telegram Bot Token 和 Chat ID，**当** 点击"发送测试消息"，**则** 指定的 Telegram 聊天在 5 秒内收到测试消息。

### 边界情况

- 站点的 API Key 被撤销或过期时怎么办？系统应显示认证错误，而非无限重试。
- 站点返回非预期的余额单位（不是 USD/CNY/Token）时怎么办？系统应显示原始值和单位，对无法识别的单位跳过阈值比较。
- 注册了 50+ 个站点，轮询周期开始时怎么办？系统应限制并发请求数，避免网络资源耗尽或触发上游限流。
- 启动时数据库文件损坏或丢失怎么办？系统应以空表重新创建数据库，并记录警告日志。
- 发送告警时 Telegram API 不可达怎么办？系统应重试一次，然后记录失败日志并继续——不应阻塞轮询周期。
- 用户为同一站点设置了重叠或无序的阈值（如 [$10, $50]）怎么办？系统应自动按从高到低排序，确保告警按正确顺序触发。
- 通过 Telegram `/balance` 触发实时查询时，若查询耗时较长怎么办？系统应先回复"正在查询中..."的即时反馈，查询完成后再发送结果。

## 需求 *(必填)*

### 功能需求

- **FR-001**: 系统必须允许用户注册上游站点（名称、URL、API Key）。
- **FR-002**: 系统必须支持两种余额查询认证方式：Bearer Token（HTTP Header）和 URL Key（查询参数）。
- **FR-003**: 系统必须通过依次探测已知端点（OpenAI 兼容、sub2api、JWT 认证、NewAPI Token）来自动检测上游站点的 API 格式。
- **FR-004**: 系统必须在首次成功检测后缓存每个站点的 API 格式，后续查询直接复用。
- **FR-005**: 系统必须以用户可配置的间隔（最小 5 分钟，默认 30 分钟）定期轮询所有启用告警的站点。
- **FR-006**: 系统必须在启动后立即执行一轮全量余额检查。
- **FR-007**: 系统必须支持每个站点设置多个告警阈值（有序列表，从高到低），每个阈值在余额首次低于时触发一次 Telegram 通知。
- **FR-008**: 系统必须跟踪每个阈值的触发状态：已触发的阈值不再重复告警；当余额回升到某阈值之上时，该阈值恢复为待触发状态。
- **FR-009**: 系统必须提供 Telegram Bot，响应 `/balance`、`/balance <名称>`、`/refresh`、`/status` 和 `/help` 命令。
- **FR-010**: 当用户通过 Telegram Bot 发送 `/balance` 命令时，系统必须对所有站点发起实时余额查询并返回最新结果，而非仅返回缓存数据。
- **FR-011**: 系统必须提供 Web 仪表盘，显示所有站点的当前余额、状态和最后检查时间。
- **FR-012**: 系统必须允许用户编辑和删除现有站点，并调整每站点的多级告警阈值。
- **FR-013**: 系统必须允许用户通过 Web 面板配置全局设置（轮询间隔、Telegram 凭证）。
- **FR-014**: 系统必须在轮询周期中限制并发余额查询数，以防止网络资源耗尽（默认上限：10 个并发请求）。
- **FR-015**: 系统必须对单个余额查询请求设置 10 秒超时。
- **FR-016**: 系统必须在重启后持久化所有站点配置、设置和阈值触发状态。

### 关键实体

- **站点 (Site)**: 一个上游 API 中转站提供商。由名称和 URL 标识。拥有 API Key 用于认证、当前余额（含单位）、探测到的 API 类型、状态，以及一组有序的告警阈值列表（从高到低）。
- **阈值 (Threshold)**: 隶属于某个站点的告警触发点。包含阈值金额和触发状态（待触发/已触发）。当余额首次低于该金额时标记为已触发并发送通知；当余额回升到该金额之上时重置为待触发。
- **设置 (Setting)**: 系统级配置参数，以键值对存储（轮询间隔、Telegram Bot Token、Telegram Chat ID）。

## 成功标准 *(必填)*

### 可量化成果

- **SC-001**: 用户添加新站点后，15 秒内可在仪表盘上看到其余额。
- **SC-002**: 系统能正确探测并查询所有 4 种支持的 API 格式，无需手动逐站配置。
- **SC-003**: 余额穿越某个阈值后，对应的 Telegram 告警在 60 秒内送达。
- **SC-004**: 系统可同时监控 50 个站点，不出现轮询漏检或响应时间下降。
- **SC-005**: 零误报：每个阈值仅在余额首次低于时触发一次，不重复发送；回升后正确重置。
- **SC-006**: 通过 Telegram `/balance` 命令查询时，所有站点的实时余额在 30 秒内返回。
- **SC-007**: Web 仪表盘在 3 秒内加载并显示所有站点状态。
- **SC-008**: 系统可连续运行 30 天无需人工干预或崩溃。
- **SC-009**: 新用户可在 10 分钟内完成部署并添加第一个站点。

## 假设条件

- 用户至少拥有一个 AI API 中转站，且其 API Key 有权限查询 billing/余额端点。
- 用户拥有 Telegram 账号，能通过 BotFather 创建 Telegram Bot 用于接收通知。
- 系统部署在具有出站 HTTPS 访问权限的基础设施上（需同时访问 API 中转站和 Telegram API）。
- API 中转站使用 4 种主流框架之一（New API / One API、sub2api、JWT 认证、NewAPI Token 格式）。自定义或私有格式不在 MVP 范围内。
- 这是一个单用户系统——没有多租户、用户认证或基于角色的访问控制。访问安全由部署环境/网络层面处理。
- API 中转站返回的余额单位在同一站点内保持一致（不会在不同查询间切换 USD 和 CNY）。
- MVP 阶段 API Key 以明文存储，加密存储是后续增强功能。
- 每个站点的告警阈值数量通常为 2-5 个，系统不需要支持超大量阈值的场景。

## 不在范围内

- 多用户支持、用户认证或基于角色的访问控制
- 余额历史记录或趋势可视化
- 基于 Cookie 的余额查询认证
- 站点配置的批量导入/导出
- Telegram 以外的通知渠道（Slack、邮件、Webhook 等）
- 消费速率预测或"预计 N 天耗尽"计算
- 超出 4 种内置探测器的自定义/私有 API 格式支持
- 移动端原生 App
- API Key 静态加密存储
